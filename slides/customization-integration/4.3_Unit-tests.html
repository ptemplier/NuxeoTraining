# Nuxeo training - Unit tests
### Why unit testÂ ? #### Code faster - No need to restart your instance every time #### Code coverage - Avoid regressions - The sooner a bug is detected, the easier it can be fixed #### Refactoring friendly - Make sure the logic remains working after code changes
### Unit testing #### Nuxeo provides a custom Nuxeo JUnit4 runner that: - Allows to deploy Nuxeo bundles, components and extension points - Launches a separate test Nuxeo document repository, reset after every test - Make sure: - you deploy the needed bundles, - initialize your documents, - and create the needed users before running your tests ! #### Unit test classes - Should be located in src/test/java - Should start with `Test`, for instance TestBookType.java
### Deploying components #### XML contributions @Deploy({"bundle.name:/OSGI-INF/file.xml"}) #### Bundles @Deploy({"an.existing.bundle","another.one"}) - Every bundle has a unique ID - In the @Deploy: use the Bundle-SymbolicName of your bundle's src/main/resources/META-INF/MANIFEST.MF file 
Bundle-ActivationPolicy: lazy
Bundle-ClassPath: .
Manifest-Version: 1.0
Bundle-Name: best-books
Bundle-RequiredExecutionEnvironment: JavaSE-1.7
Bundle-Vendor: Nuxeo
Bundle-Version: 1.0.qualifier
Bundle-ManifestVersion: 2
Bundle-SymbolicName: best-books
### Deploying bundles #### Which bundles should I deploy ? - Probably many ! - To help deploying the common bundles, Nuxeo uses Junit 4's FeatureRunner #### What's a feature ? - A class extending the SimpleFeature class - Annotated just as a test class - Containing an @Deploy annotation, with the layer's bundles #### How can I use it ? - Add the following annotations to your test class: 
@RunWith(FeaturesRunner.class)
@Features(FeatureName.class)
### Available features #### Which features are available ? - RuntimeFeature - CoreFeature - PlatformFeature - AutomationFeature #### What do they bring ? - A feature exists for most of the platform's layers - The UI layer is not unit testable, for this we use integration tests
### Practice - Nuxeo IDE #### Annotations - Create a test class named `TestStudioBundle` - Make it run with the RuntimeFeature - Run the class as a JUnit test class (right click, run as, JUnit test) - What happens ?

### Using test methods - Every test method needs to be annotated with @Test - Your method should contain one or more assertions - To declare an assertion, use JUnit's Assert.assert... methods, ex: assertNotNull(MyObject)
### Practice - Nuxeo IDE #### Unit test methods - In your class, create a test method named `shouldDeployNuxeoRuntime()` returning void - Use Framework.getRuntime() (org.nuxeo.runtime.api) and place it into a runtime variable - Assert that the runtime variable is not null - Launch your test - What happens ?
## Questions?
### Nuxeo services injection #### How can I work on documents from my unit tests ? - To work on documents, you need a CoreSession #### A CoreSession ? - Is different from a HTTP session - Is sort of a special service - Contains methods to work on your repository (create, save... documents) - Needs the CoreFeature or more #### How do I call it in a unit test ? - CoreSession can be injected with Google Guice injection mechanism 
@Inject CoreSession documentManager;
### Nuxeo Core API - Useful objects #### CoreSession - Called when you need to access the repository directly
ex: Store / retrieve documents in the database #### DocumentModel - Stores document information in memory and transmits it to the CoreSession for storage - Called when you need to access and / or update the document properties - A document is an instance of a DocumentModel object #### NuxeoPrincipal - Object containing user information - Called when you need to create / update / delete a user or a user group
### API basics #### Creating a document DocumentModel doc = coreSession.createDocumentModel(parentPath, id, typeName);
Creates the DocumentModel object.
This method calls the EmptyDocumentCreated event we have seen previously in Studio.
doc.setPropertyValue(xpath, value);
Set a metadata using its xpath (schemaprefix:metadata) doc = coreSession.createDocument(doc);
Create the document in the repository.
This method calls both AboutToCreate and DocumentCreated events.
coreSession.save();
Flushes the information. 
// Create document in memory
DocumentModel doc = coreSession.createDocumentModel("/", "myDocName", "File");

// Set some metadata values
doc.setPropertyValue("dc:title", "No coffee, no code");
doc.setPropertyValue("dc:description", "Understood ?");

// Save in database
doc = coreSession.createDocument(doc);

// Flush
coreSession.save();
Make sure you put your doc in a variable to get it updated ! Events will be called by these methods.
### API basics #### Fetching a document DocumentModel doc = coreSession.getDocument(docRef);
#### 3 ways to get a DocumentRef from a document - Ref: DocumentRef docRef = doc.getRef(); - Id: DocumentRef idRef = new IdRef("someDocumentID"); - Path: DocumentRef pathRef = new PathRef("someDocumentPath"); #### Getting a metadata Serializable docMetadata = doc.getPropertyValue(xpath);
// Get a document
doc = coreSession.getDocument(doc.getRef());

// Get the title and description
String docTitle= doc.getTitle();
String docDescription = doc.getPropertyValue("dc:description");
### API basics #### Updating a document DocumentModel doc = saveDocument(doc);
Save a doc from its model. doc.setPropertyValue(String xpath, Serializable value);
Update a metadata using its xpath (schemaprefix:metadata) 
doc.setPropertyValue("dc:title", "My new title");
doc = coreSession.saveDocument(doc);
coreSession.save();
### Practice - Nuxeo IDE #### Document creation and modification - Deploy your Nuxeo Studio bundle (In Studio, bundle symbolicname is in the Settings and versioning > project settings menu > manifest tab) - Inject a CoreSession (make sure you deploy the AutomationFeature) - Create a unit test named `shouldHaveAPublicationDate()` that: - Creates a `Book` document located in the root directory - Checks that the book's publication date is not null - Create a unit test named `shouldHaveCorrectTitle()` that: - Creates a `Book` document located in the root directory - Sets a title to the Book: `Da Nuxeo code` - Saves the book in the repository - Fetches the book from the repository and assert that the book's title is equal to what you expected

### Testing existing automation chains - Use AutomationServiceÂ  - In JUnit4, Nuxeo services are injected with Google Guice @Inject annotation 
@Inject AutomationService automationService
- Or 
AutomationService automationService = Framework.getLocalService(AutomationService.class)
(works also outside of a unit test) - Create an OperationContext with the CoreSession and a document input and call the operation with: 
AutomationService automationService = Framework.getLocalService(AutomationService.class);
OperationContext ctx = new OperationContext();
ctx.setCoreSession(session);
ctx.setInput(myInputDocument);
automationService.run(ctx, "chainId");
