Nuxeo Platform - Maintaining Your Application

### Context
Our production server is ready. Cool! Now it's time to think about what comes next: backing up data, applying hotfixes and upgrading.
### Maintenance - Using nuxeoctl #### nuxeoctl Script - Can be called to maintain the application while it is stopped - Otherwise the same commands can be done through the interface in the admin tab - Is extremely useful for automating maintenance - Used to handle marketplace packages, hotfixes, application start and stop #### Main nuxeoctl Commands - nuxeoctl help Lists all available commands - nuxeoctl configure To test configuration templates - nuxeoctl mp-hotfix --accept yes Quietly install all available hotfixes - nuxeoctl mp-list Lists local marketplace packages - nuxeoctl mp-install Install a marketplace package
### Maintenance - Using the Web UI #### Update Center (Admin tab, Update Center menu) - Used to install, upgrade, remove marketplace packages or hotfixes ##### Available Tabs - Nuxeo Studio: to install Studio releases - Nuxeo software updates: to apply hotfixes - Private packages: private marketplace packages you have access to with your account (Studio releases, privately shared marketplace packages between customers) - Packages from Nuxeo Marketplace: all marketplace packages, including the ones available from the marketplace - Local packages: already downloaded / installed marketplace packages, or manually imported.
### Applying Hotfixes
Apply hotfixes on your application.
- Either using the nuxeoctl if your application is stopped - Or using the Web UI if the application is running

### Orphaned Binaries Management #### Deleting Orphaned Binaries ##### Binaries are not deleted immediately on document deletion, because: - Other documents could still reference it ! - And it is costly to check every document for that ##### They can be deleted... - Manually in Admin, System Information, Repository Binaries - Through a groovy script - Using some Java code that can be triggered through a scheduled task or any kind of event 
import org.nuxeo.ecm.core.storage.sql.management.RepositoryStatus;

RepositoryStatus status = new RepositoryStatus();
if (!status.isBinariesGCInProgress()) {
    status.gcBinaries(true);
}

ADMINDOC/Backup+and+Restore
###Backup/restore #### Hot backup ##### Do a database backup first - SQL dump can be done while the database is running with postgreSQL - elasticsearch index is fast to rebuild if needed - Back it up only if it is really big - It has to be backed up at the same time than the db for consistency ##### Save the Filesystem second (binary store)

####VCS ensures backup consistency - No risk of file being in process of update - No risk of file referenced by the DB that has been removed from the filesystem
###Sample with PostgreSQL ####Data Backup - Save db: pg_dump dbname | gzip > filename.gz - Save data directory (archive, rsync or use your own filesystem backup tool) - Save log directory ####Conf Backup - Save nuxeo.conf file and custom templates. ####Data Restore - createdb dbname - Import db (see also pg_restore): gunzip -c filename.gz | psql dbname - Restore data directory ####Upgrade - Backup data and configuration (see above) - Deploy new Nuxeo instance and reuse previous instance configuration
[ADMINDOC/Upgrading+your+Nuxeo+Version](http://doc.nuxeo.com/x/rgc7)
###Upgrade distribution ####Before any upgrade - Read the Upgrade Notes - Backup database - Backup the data directory - Backup configuration and custom templates - Backup log directory ####Upgrade - Deploy new Nuxeo - Adapt your configuration - Start Nuxeo
